本文件逐代码讲解 d:\deep-agents\src\standard_document_assistant 下的标准文档助手， 配套 docs.langchain 官方 Deep Agents 文档（overview / customization / human-in-the-loop / context-engineering / subagents / backends / skills）， 并指出如何自定义 tools / skills / subagents，以及它们在哪里、如何被调用。

1. 一句话概览

本项目在做什么

标准文档助手（standard-document-assistant）是一个面向国标 / 行标 PDF / Word 的智能体，覆盖 解析 元数据抽取 内容审核 格式审核 草稿生成 五大业务。 它通过 deepagents.create_deep_agent 工厂构造。

Deep Agents 范式

官方称 Deep Agents 为 "agent harness"：在 LangGraph 工具调用循环之上， 内置 任务规划、文件系统、子代理、 长期记忆、技能、HITL、上下文工程 7 大能力。 本项目把 harness 适配到 GB 标准文档审核业务。

2. Deep Agents 框架核心能力

① 任务规划 TodoList

内置 write_todos。

本项目在 prompts.py:MAIN_SYSTEM_PROMPT 强制要求主 Agent 在复杂任务前先调用 write_todos。

② 文件系统 Backend

State/Store/Filesystem/Composite。

本项目使用 CompositeBackend，把 /memories/、/skills/、 /workspace/ 三类虚拟路径路由到不同后端。

③ SubAgent 委派

隔离上下文、并行、专业化。

本项目注册了 parser / extractor / reviewer / research / writer 5 个 SubAgent， 主 Agent 通过内置 task 工具按需委派。

④ 长期记忆

StoreBackend + LangGraph Store。

本项目通过 seed_memory_store 启动注入 AGENTS.md / preferences.md / project-notes.md。

⑤ Skills 按需加载

SKILL.md frontmatter 渐进披露。

本项目在 skills/ 目录下放 4 个 SKILL.md，启动只读 frontmatter， 任务命中 description 才加载正文。

⑥ HITL 人工审批

interrupt_on。

本项目对写文件、命令执行、记忆更新、MinerU 解析、审核等敏感工具配置 ["approve", "edit"] 决策粒度。

⑦ 结构化输出

response_format。

本项目通过 response_format=AgentResult 强制主 Agent 输出 summary / artifacts / quality_warnings / next_actions / download。

⑧ 上下文工程

system + memory + skills + middleware。

本项目把 LangGraph context_schema 用于子图透传 trace_id / cover_metadata_hint / quality_strict。

⑨ LangGraph 持久化

checkpointer / store。

本地 MemorySaver + InMemoryStore；部署由 LangGraph Server 注入。 langgraph_server 参数切换。

def build_standard_document_agent(*, strict_model: bool = False, langgraph_server: bool = False):
    from deepagents import create_deep_agent  # 框架入口

    checkpointer = None

        from langgraph.store.memory import InMemoryStore

        checkpointer = MemorySaver()

6.3 关键设计点

  • 本地 / 部署双模式：langgraph_server 开关决定是否让平台托管 checkpointer / store。
  • 结构化输出：response_format=AgentResult（schemas.py）。
  • Skills 注入：skills=["/skills/"] 必须使用虚拟路径，Deep Agents 运行时按 frontmatter 描述按需加载。
  • HITL 分层：主 Agent 层只覆盖"主 Agent 自身直调"的工具；subagent 内的工具由 build_subagents 内部单独声明。

通用规则

  • SubAgent 是无状态的：每次 task(...) 都从干净上下文启动；指令需一次给完整。
  • SubAgent 不继承主 Agent 的 Skill，必须在 spec 内显式声明 skills=[...]。
  • SubAgent 仍可调用 Deep Agents 内置工具（write_todos、ls/read_file/write_file），由框架自动注入。
  • HITL interrupt_on 可在 SubAgent 层独立声明，避免与主 Agent 重复。

Skill 在哪里被调用？

  1. 主 Agent 启动时 create_deep_agent(skills=["/skills/"])，Deep Agents 框架读取每个 SKILL.md 的 frontmatter，只把 name+description 注入 system prompt。
  2. 当 SubAgent 描述 / 用户请求匹配某 skill 的 description，Deep Agents 才会按需读取该 SKILL.md 正文。
  3. SubAgent spec 内 skills=["/skills/standard-parsing"]（agent.py） 是 绑定关系：告诉主 Agent "这个 SubAgent 适用哪个 skill"，避免主 Agent 把 skill 描述混进自己 prompt。

Skill 目录结构遵循 Deep Agents Skills 规范：

skills/<skill-name>/SKILL.md + 可选 references/

三步新增一个 Skill：

  1. 新建目录：skills/standard-translation/
  2. 写 SKILL.md（frontmatter 必填 name + description）
  3. 在 agent.py:build_subagents 把 "/skills/standard-translation" 加到目标 SubAgent 的 skills=[...] 列表

模板

---
name: standard-translation
description: |
  将 /workspace/output/drafts/*.md 中文标准草稿翻译为英文 Markdown。
  触发条件：用户输入"中英双语" / "英文版" / "translate to English"。
  强约束：保留条款编号、不得编造引文、不得替换专有标准名。
---

# Standard Translation

## 何时使用
1. 用户要求把已生成的 .md 草稿翻译成英文。
2. 已经存在 /workspace/output/drafts/<stem>.md。

## Instructions
1. 调用 write_file 写入 /workspace/output/drafts/<stem>.en.md。
2. 不得修改条款编号。
3. 不确定术语保留原文 + 译文括注。

## 失败处理
- 缺原文：先 task 委派 writer 生成中文草稿。

常见错误

  • frontmatter 缺 name 或 description：Deep Agents 无法发现该 skill。
  • description 过于笼统（如“帮助用户”）：主 Agent 难以判断何时加载；应具体描述触发条件。
  • 把大文件直接塞进 SKILL.md 正文：应放 references/，由 Agent 按需读取。

  • 把 runtime 写进 args_schema：会触发 PydanticInvalidForJsonSchema: core_schema.CallableSchema；必须用 InjectedToolArg 标记。

23. 两个子图的设计与代码（精读）

本节完整精读 graphs/metadata_extraction/ 和 graphs/standard_review/ 两个子图。两者都遵循「StateGraph + context_schema + 节点级 RetryPolicy + get_stream_writer」的同一套设计骨架， 但拓扑结构、状态结构、回流控制完全不同。

23.1 共同设计骨架

① StateGraph + context_schema

两图都使用 StateGraph(State, context_schema=Context)，其中 State 是 TypedDict, total=False 描述的状态字典， Context 是横切关注点（trace_id / job_id / cover_metadata_hint / quality_strict / parent_tool_call_id），通过 graph.invoke(..., context=...) 由 Tool 层透传，节点内用 runtime.context 读取，避免污染 state。

依据 LangGraph Runtime context 与 Deep Agents Runtime context。

② 节点级 RetryPolicy

对调用 LLM 的节点（run_langextract / judge_rules / aggregate）挂 RetryPolicy(max_attempts=2, jitter=True, retry_on=(RequestException, RuntimeError))。 只重试网络 / 协议层瞬态错误，不重试 LLM 输出解析错误，避免无效消耗。

依据 LangGraph 异常处理："Only failing branches are retried"。

③ get_stream_writer 推送 SSE

所有节点用 get_stream_writer() 推送命名空间事件；与 MinerU 的 mineru.*、审核的 review.* 共同形成 <domain>.<stage> 统一规范。 失败容忍：图外调用（单测 / Tool 直调）RuntimeError 时静默忽略。

④ checkpointer 不显式挂

依靠主图 compile(checkpointer=MemorySaver()) 自动传播到子图，命名空间由 thread_id 隔离。依据 Use in subgraphs。

节点精读（nodes.py）

1) load_markdown · 读源
优先使用 state.markdown（由 Tool 直接传入，避免重复 IO）；否则从 source_virtual_path 解析到 host 路径后读 UTF-8。 路径解析走 resolve_workspace_read_path(allowed_roots=[UPLOADS_DIR, OUTPUT_DIR], suffixes={.md/.markdown/.txt})， 把 /workspace/ 虚拟路径安全转回 host 路径并校验后缀。
2) slice_scope · 范围截取 + 字节上限
调用 slice_metadata_scope(text, mode)： "full" 取全文，"metadata" 用正则 (?im)^\s*#{1,6}\s*(?:第\s*4\s*章|4)(?!\s*[\.．]\s*\d)\s*… 截到第 4 章前（聚焦封面 / 前言 / 2-3 章）。 截取后按 config.metadata_extraction.scoped_text_max_bytes 二次截断（UTF-8 编码后按字节），避免 LLM 超限。 推送 meta.scoped 流事件。
3) run_langextract · 唯一一次外部 LLM 调用
  • 用 @traceable(run_type="chain", name=...) 包装 _traced_run_extraction，让 LangSmith 看到独立 run。
  • 环境变量 METADATA_MODEL_ID 或 MODEL_ID 覆盖 model label（仅用于流事件展示）。
  • 推送 meta.extraction.start / meta.extraction.end / meta.extraction.error。
  • 挂 _LANGEXTRACT_RETRY_POLICY（详见骨架 ②）。
  • 失败时设置 status="failed"，后续节点看到 status==failed 短路返回。
4) aggregate_fields · 字段聚合 + 封面 hint 兜底
调用 build_extraction_result(result, source_label)：单值字段按"最早 start_pos 出现"取 1 个，多值字段按"start_pos + 文本"排序去重；推断 标准性质（GB 强 / GB/T 推 / GA 推 / GB/Z 指导性）和 制修订（有代替标准号=修订）。

关键：从 runtime.context.cover_metadata_hint 读封面 hint（Deep Agents best practice），与 state.cover_metadata_hint 双向兜底： ICS / CCS / 标准层级 / 标准号 / 代替标准号 5 个高置信字段若 langextract 没抽出，会被 hint 覆盖。 推送 meta.aggregated（含单值 / 多值字段数、quality_warnings 计数）。

关键设计：不再自己 StandardMetadataExtraction.model_validate(...)， 而走 tools.validation.validate_output_schema(aggregated, schema_name="StandardMetadataExtraction")， 与 Agent 工具调用入口保持完全一致的 schema 名 / 错误格式（依据 module_architecture_review.md §4.3 item 5）。

strict 标志优先取 runtime.context.quality_strict，兜底 config.metadata_extraction.strict_validation：
  • 严格模式：校验失败 → status="failed"，后续节点短路。
  • 宽松模式：保留 langextract 原始聚合结果，仅加 quality_warnings。

解析算法（parse_markdown_text）

  1. 定位目次：_find_toc_start 扫前 ~30 行找 "目次" / "目录" 标题。命中后： [0, toc_idx) 是封面，[toc_idx, body_start) 是目次。
  2. 定位正文起点：_is_body_start_line 同时要求 (a) 标题或编号格式 且 (b) 章节名被分类器识别（foreword / scope / normative_references 等 9 类之一）， 并且不是目次条目（首列无空格、宽度 < 30）。
  3. 章节切分：_split_body_chunks 找所有 _is_chunk_heading 行号； 连续两个 heading 之间的内容属前一个 chunk。每 chunk 计算 line_start / line_end，分类到 scope。
  4. 章节名 → scope：_classify_heading 基于中文字面（"范围"→scope、"规范性引用文件"→normative_references、"术语和定义"→terms_definitions 等）。 未知 scope 归为 other_body；appendices（附录 / 索引 / 参考文献）一类 不被收进 body_text，单独进 appendix_text / index_text / references_text / end_text。
  5. body_text 过滤：_build_body_text 只保留 {scope, normative_references, terms_definitions, symbols_abbreviations, other_body}， 且只保留 order ≥ first_scope_order 的 chunks（确保不含前言、引言）。
  6. 结果装配：返回 ParsedMarkdownDocument，每个 scope 字段、section_chunks（带行号）、行号区间都填好。
章节分类 _classify_heading 完整规则（doc_parser.py）

  • 判定关键词："前言"→foreword、"引言"→introduction、"范围"→scope、"规范性引用"/"引用文件"→normative_references、"术语"/"定义"→terms_definitions、"符号"/"缩略语"→symbols_abbreviations、"附录"→appendix、"参考文献"→references、"索引"→index

2.3 filter_rules_for_partial_mode · 规则过滤

def filter_rules_for_partial_mode(rules, partial_mode, active_keys, *, include_full_document_scope=True):
    """按分析模式 × active_keys 三维过滤：local/cross_section 仅当 target_scopes 与 active_keys 交集非空。"""
    if partial_mode == "full_document":
        return list(rules)            # full 模式不筛
    keys = set(active_keys)
    filtered = []
    for rule in rules:
        if rule.analysis_mode == "local" or rule.analysis_mode == "cross_section":
            if rule.target_scopes and keys.intersection(rule.target_scopes):
                filtered.append(rule)
        elif rule.analysis_mode == "full_document":
            if include_full_document_scope:
                filtered.append(rule)
        else:
            filtered.append(rule)
    return filtered

24.5 第 4 层：检索 retrieve.py

完整算法流程：

def retrieve_rules(state, runtime=None) -> dict:
    # 1. 加载知识库（FAISS 优先，回退 JSON）
    kb, kb_meta = load_knowledge_base(config, force_rebuild=force)
    rules = filter_content_audit_rules(kb.rules)   # 排除 format / format_only

    # 2. 确定 active_keys
    active_keys = list(state.get("active_scope_keys") or [])
    if partial_mode == "full_document":
        active_keys = [13 个标准 scope]
    if requested := normalize_scope_keys(state.get("target_scopes")):
        active_keys = [k for k in active_keys if k in set(requested)] or active_keys

    # 3. 对每个 active_key 检索 top_k
    section_rules, section_rule_dicts, retrieval_trace = [], [], []
    seen_ids: set[str] = set()
    for scope_key in active_keys:
        hits = kb.search(scope_key, scope=scope_key, top_k=top_k, index_dir=index_dir)
        for hit in hits:
            if hit.rule.chunk_id in seen_ids:               # 去重
                continue
            if hit.rule.analysis_mode == "full_document":   # full 规则单独入 full_document_rules
                continue
            if hit.rule.target_scopes and scope_key not in hit.rule.target_scopes:
                if hit.rule.scope != scope_key:
                    continue
            seen_ids.add(hit.rule.chunk_id)
            section_rules.append(hit.rule)
            section_rule_dicts.append(_rule_to_dict(hit.rule, matched_scope=scope_key))
        retrieval_trace.append({"scope": scope_key, "top_k": top_k,
                                "rule_ids": [h.rule.chunk_id for h in hits],
                                "query": f"scope:{scope_key}"})

    # 4. full_document 规则单独收集（所有 analysis_mode=="full_document" 的规则都进）
    full_doc_rules = [r for r in rules if r.analysis_mode == "full_document"]

    return {
        "section_rules": section_rule_dicts,
        "full_document_rules": [...],
        "section_rule_objects": section_rule_dicts,         # 留给 judge 反序列化

def _choose_strategy(self, rule: RuleItem, context_len: int) -> str:
    if rule.analysis_mode == "full_document":   return "full_document"
    if rule.analysis_mode == "cross_section":  return "cross_section"
    if rule.analysis_mode == "deterministic":  return "skip"     # 确定性规则不走 LLM

async def _judge_with_sem(self, sem, plan, *, file_name, trace_id) -> JudgeOutcome:
    async with sem:
        writer = get_stream_writer()                 # 推 "scope_progress" 事件
        writer({"type": "scope_progress", "rule": plan.rule.chunk_id, "strategy": plan.strategy})
        try:
            payload = await self._invoke_strategy(plan)
        except Exception as exc:
            writer({"type": "judge_error", "rule": plan.rule.chunk_id, "error": str(exc)})
            return JudgeOutcome(..., issue=self._fallback_issue(plan, file_name=trace_id, error=str(exc)), raw="")
        return JudgeOutcome(rule=plan.rule, strategy=plan.strategy, issue=self._build_issue(plan, payload, file_name=file_name), raw=json.dumps(payload, ensure_ascii=False))

5.5 _build_issue · 从 LLM JSON 到 AuditIssue

def _build_issue(self, plan, payload, *, file_name) -> AuditIssue | None:
    if not payload:                                return _fallback_issue(status="insufficient_context")
    if bool(payload.get("pass")):                  return None   # 通过就不产生 issue
    confidence = float(payload.get("confidence", 0.0))
    status = "fail" if confidence > self.config.low_confidence_floor else "insufficient_context"
    return AuditIssue(
        issue_id=f"{plan.rule.chunk_id}-{file_name}",
        severity=str(payload.get("severity_level") or "中度"),
        status=status,                              # fail / insufficient_context
        expected=plan.rule.content,                 # 期望：规则原文
        actual=str(payload.get("actual") or "..."),
        evidence_text=str(payload.get("evidence_text") or ""),
        source_ref=plan.rule.source_ref,
        suggestion=str(payload.get("suggestion") or ""),
        confidence=confidence,
        llm_reasoning=str(payload.get("reasoning") or ""),
    )

9.1 聚合 aggregate.py

def aggregate(state, runtime=None) -> dict:
    issues = state.get("issues") or []
    statuses = Counter(item.get("status", "") for item in issues)   # fail / warn / pass / insufficient
    severities = Counter(item.get("severity", "") for item in issues)
    tracks = Counter(item.get("audit_track", "") for item in issues)
    scope_buckets = {}
    for issue in issues:
        key = f"{issue.get('audit_track') or 'content_llm'}::{issue.get('scope') or 'full_document'}"
        bucket = scope_buckets.setdefault(key, {"track": ..., "scope": ..., "total": 0, "fail": 0, "warn": 0, "pass": 0, "insufficient": 0, "issues": []})
        bucket["total"] += 1
        status = issue.get("status", "")
        bucket[status if status in {"pass","warn","fail","insufficient_context"} else "warn"] += 1
        bucket["issues"].append(issue.get("issue_id", ""))
    summary = {
        "total_issues": len(issues),
        "failed": statuses.get("fail", 0),
        "warn": statuses.get("warn", 0),
        "insufficient_context": statuses.get("insufficient_context", 0),
        "by_severity": dict(severities),
        "by_track": dict(tracks),
        "by_scope": {k: {k: v for k, v in b.items() if k != "issues"} for k, b in scope_buckets.items()},
    }
    final_status = "success" if not state.get("errors") else "failed"
    return {"scope_summary": scope_buckets, "aggregate_summary": summary, "status": final_status, "final_status": final_status, ...}

24.11 关键设计决策一览

决策理由
LLM 异常不重试RetryPolicy.retry_on 只列 (RequestException, RuntimeError)；JSON 解析失败通过 safe_json_loads + _empty_result 转 _fallback_issue
章节缺失 → warn（不 fail）"应当存在但缺失" 不强制阻断，但要在 trace + report 里可见
扩域最多 N 轮防止 widen → reload → judge 死循环；N 默认 2，由 config.max_review_rounds 配
LLM 摘要可选config.enable_audit_summary=False 时完全跳过；LLM 不可用时降级到 _offline_summary（按 severity × scope 统计）
双通道事件emit_event 同时写 state["trace_events"] + 推 review.* SSE；不破坏 reviewer subagent / report.json / trace.json 的旧消费者
三后端分层（FAISS / JSON / memory）faiss-cpu 装上走原路径，缺包回退到 JSON，再退到内存；保证 CI / 离线环境都能跑
确定性 + LLM 双轨格式轨 5 项全是确定性正则检查；内容轨先 LLM 再叠 scope 覆盖；任何环节失败都有"依据不足"issue，不会漏报
LLM Judge 多策略不同 analysis_mode 用不同 prompt + 上下文窗口长度；策略选择纯函数 _choose_strategy，可单测
