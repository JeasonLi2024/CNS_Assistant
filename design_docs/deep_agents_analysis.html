标准文档助手 · Deep Agents 框架代码精读

本文件逐代码讲解 d:\deep-agents\src\standard_document_assistant 下的标准文档助手， 配套 docs.langchain 官方 Deep Agents 文档（overview / customization / human-in-the-loop / context-engineering / subagents / backends / skills）， 并指出如何自定义 tools / skills / subagents，以及它们在哪里、如何被调用。

1. 一句话概览

本项目在做什么

标准文档助手（standard-document-assistant）是一个面向国标 / 行标 PDF / Word 的智能体，覆盖 解析 元数据抽取 内容审核 格式审核 草稿生成 五大业务。 它通过 deepagents.create_deep_agent 工厂构造。

Deep Agents 范式

官方称 Deep Agents 为 "agent harness"：在 LangGraph 工具调用循环之上， 内置 任务规划、文件系统、子代理、 长期记忆、技能、HITL、上下文工程 7 大能力。 本项目把 harness 适配到 GB 标准文档审核业务。

2. Deep Agents 框架核心能力

① 任务规划 TodoList

内置 write_todos。

本项目在 prompts.py:MAIN_SYSTEM_PROMPT 强制要求主 Agent 在复杂任务前先调用 write_todos。

② 文件系统 Backend

State/Store/Filesystem/Composite。

本项目使用 CompositeBackend，把 /memories/、/skills/、 /workspace/ 三类虚拟路径路由到不同后端。

③ SubAgent 委派

隔离上下文、并行、专业化。

本项目注册了 parser / extractor / reviewer / research / writer 5 个 SubAgent， 主 Agent 通过内置 task 工具按需委派。

④ 长期记忆

StoreBackend + LangGraph Store。

本项目通过 seed_memory_store 启动注入 AGENTS.md / preferences.md / project-notes.md。

⑤ Skills 按需加载

SKILL.md frontmatter 渐进披露。

本项目在 skills/ 目录下放 4 个 SKILL.md，启动只读 frontmatter， 任务命中 description 才加载正文。

⑥ HITL 人工审批

interrupt_on。

本项目对写文件、命令执行、记忆更新、MinerU 解析、审核等敏感工具配置 ["approve", "edit"] 决策粒度。

⑦ 结构化输出

response_format。

本项目通过 response_format=AgentResult 强制主 Agent 输出 summary / artifacts / quality_warnings / next_actions / download。

⑧ 上下文工程

system + memory + skills + middleware。

本项目把 LangGraph context_schema 用于子图透传 trace_id / cover_metadata_hint / quality_strict。

⑨ LangGraph 持久化

checkpointer / store。

本地 MemorySaver + InMemoryStore；部署由 LangGraph Server 注入。 langgraph_server 参数切换。

6.1 持久化与模型初始化

d:\deep-agents\src\standard_document_assistant\agent.py · L107-L143
def build_standard_document_agent(*, strict_model: bool = False, langgraph_server: bool = False):
    from deepagents import create_deep_agent  # 框架入口
    ensure_project_directories()              # 创建 workspace/、memories/、skills/ 子目录
    config = load_config()                     # 读 config.yaml + .env
    model = build_qwen_model(config.primary_model, strict=strict_model)

    # 持久化：本地 MemorySaver + InMemoryStore；部署态由平台注入
    store = None
    checkpointer = None
    if not langgraph_server:
        from langgraph.checkpoint.memory import MemorySaver
        from langgraph.store.memory import InMemoryStore
        store = InMemoryStore()
        seed_memory_store(store)               # 把 memories/*.md 注入 Store
        checkpointer = MemorySaver()

6.2 create_deep_agent 关键 kwargs

d:\deep-agents\src\standard_document_assistant\agent.py · L144-L177

        "backend": build_backend(langgraph_server=langgraph_server, store=store),
        "permissions": build_permissions(),                               # 路径级 read/write
        "response_format": AgentResult,                                  # Pydantic 结构化输出
    }
    if hitl_enabled(langgraph_server=langgraph_server):
        # 主 Agent 层只覆盖"主 Agent 自身直调"的工具；
        # subagent 内的工具由 build_subagents 内部声明。
        agent_kwargs["interrupt_on"] = {
            "write_file":            {"allowed_decisions": ["approve", "edit"]},
            "edit_file":             {"allowed_decisions": ["approve", "edit"]},
            "propose_memory_update": {"allowed_decisions": ["approve", "edit"]},
            "execute":               {"allowed_decisions": ["approve", "edit"]},

    if checkpointer is not None: agent_kwargs["checkpointer"] = checkpointer
    if store is not None:         agent_kwargs["store"] = store

6.3 关键设计点

  • 本地 / 部署双模式：langgraph_server 开关决定是否让平台托管 checkpointer / store。
  • 结构化输出：response_format=AgentResult（schemas.py）。
  • Skills 注入：skills=["/skills/"] 必须使用虚拟路径，Deep Agents 运行时按 frontmatter 描述按需加载。
  • HITL 分层：主 Agent 层只覆盖"主 Agent 自身直调"的工具；subagent 内的工具由 build_subagents 内部单独声明。

9. 5 个 SubAgent 总览

SubAgent 由 agent.py:build_subagents 构造（L68-L105），传入 create_deep_agent(..., subagents=...)。

parser

PDF/Word → Markdown + manifest + cover_metadata

  • 系统提示：PARSER_PROMPT
  • 绑定 Skill：/skills/standard-parsing
  • 核心工具：parse_file_with_mineru
  • HITL：parse_file_with_mineru → approve / edit
  • AGENTS.md：subagents/parser/AGENTS.md

extractor

Markdown → 国标 19 个元数据 JSON

  • 系统提示：EXTRACTOR_PROMPT
  • 绑定 Skill：/skills/standard-extraction
  • 核心工具：extract_standard_metadata、validate_output_schema
  • HITL：extract_standard_metadata → approve / edit
  • AGENTS.md：subagents/extractor/AGENTS.md

reviewer

内容轨 + 格式轨 + scope_summary + audit_summary

  • 系统提示：REVIEWER_PROMPT
  • 绑定 Skill：/skills/standard-review
  • 核心工具：parse_file_with_mineru、run_standard_review、run_format_source_review、inspect_review_rules、build_review_index、validate_review_result_schema
  • HITL：parse / run_standard_review / run_format_source_review / build_review_index → approve / edit
  • AGENTS.md：subagents/reviewer/AGENTS.md

research

参考资料梳理（仅基于现有材料）

  • 系统提示：RESEARCH_PROMPT
  • 绑定 Skill：—（无）
  • 核心工具：内置文件工具
  • HITL：受主 Agent 层 write_file/edit_file 覆盖
  • AGENTS.md：subagents/research/AGENTS.md

writer

标准文档 Markdown 草稿

  • 系统提示：WRITER_PROMPT
  • 绑定 Skill：/skills/standard-drafting
  • 核心工具：validate_output_schema + 内置 write_file
  • HITL：受主 Agent 层 write_file 覆盖
  • AGENTS.md：subagents/writer/AGENTS.md

通用规则

  • SubAgent 是无状态的：每次 task(...) 都从干净上下文启动；指令需一次给完整。
  • SubAgent 不继承主 Agent 的 Skill，必须在 spec 内显式声明 skills=[...]。
  • SubAgent 仍可调用 Deep Agents 内置工具（write_todos、ls/read_file/write_file），由框架自动注入。
  • HITL interrupt_on 可在 SubAgent 层独立声明，避免与主 Agent 重复。

SubAgent 在哪里被调用？

主 Agent 在 LangGraph 内部会自动获得 task(agent="<name>", instruction="...") 工具。 主 Agent 通过该工具委派任务。典型 prompt 节选（来自 prompts.py）：

9. 用户提交 PDF 或 Word 并要求元数据抽取时：
   先用 task 委派 parser 调用 parse_file_with_mineru，
   取得 virtual_md_path、cover_metadata、virtual_manifest_path；
   再委派 extractor 调用 extract_standard_metadata(...)。

10. 如何自定义 SubAgent

本项目在 agent.py:build_subagents 中以 dict 形式注册 SubAgent。要新增一个 SubAgent，只需三步：

步骤 1：在 build_subagents() 内追加 spec

d:\deep-agents\src\standard_document_assistant\agent.py · build_subagents()
return [
    parser_spec, extractor_spec, reviewer_spec,
    research_spec, writer_spec,
    # 👇 新增 SubAgent
    {
        "name": "translator",
        "description": "将中文标准草稿翻译为英文 Markdown，供双语发布。",
        "system_prompt": TRANSLATOR_PROMPT,
        "tools": [validate_output_schema],   # 用 LangChain @tool / StructuredTool
        "skills": ["/skills/standard-translation"],
    },
]

步骤 2：在 prompts.py 写 system prompt

TRANSLATOR_PROMPT = """你是标准文档翻译子代理。
输入：/workspace/output/drafts/*.md 中文草稿。
输出：写入 /workspace/output/drafts/*.en.md，仅返回路径与摘要。
"""

步骤 3（可选）：在 subagents/translator/AGENTS.md 落一份 managed 兼容副本

d:\deep-agents\subagents\translator\AGENTS.md
---
name: translator
description: 将中文标准草稿翻译为英文 Markdown。
---

你是标准文档翻译子代理。输入是 /workspace/output/drafts/*.md，
输出是同目录 *.en.md。不得自行改写条款编号。

步骤 4：在 prompts.py:MAIN_SYSTEM_PROMPT 加委派规则

16. 用户要求中英双语发布时，task 委派 translator 调用 write_file
    写入 /workspace/output/drafts/<stem>.en.md。
说明：SubAgent name 必须唯一，会成为 task 工具的可选值。

Skill 在哪里被调用？

  1. 主 Agent 启动时 create_deep_agent(skills=["/skills/"])，Deep Agents 框架读取每个 SKILL.md 的 frontmatter，只把 name+description 注入 system prompt。
  2. 当 SubAgent 描述 / 用户请求匹配某 skill 的 description，Deep Agents 才会按需读取该 SKILL.md 正文。
  3. SubAgent spec 内 skills=["/skills/standard-parsing"]（agent.py） 是 绑定关系：告诉主 Agent "这个 SubAgent 适用哪个 skill"，避免主 Agent 把 skill 描述混进自己 prompt。

12. 如何自定义 Skill

Skill 目录结构遵循 Deep Agents Skills 规范：

skills/<skill-name>/SKILL.md + 可选 references/

三步新增一个 Skill：

  1. 新建目录：skills/standard-translation/
  2. 写 SKILL.md（frontmatter 必填 name + description）
  3. 在 agent.py:build_subagents 把 "/skills/standard-translation" 加到目标 SubAgent 的 skills=[...] 列表

模板

---
name: standard-translation
description: |
  将 /workspace/output/drafts/*.md 中文标准草稿翻译为英文 Markdown。
  触发条件：用户输入"中英双语" / "英文版" / "translate to English"。
  强约束：保留条款编号、不得编造引文、不得替换专有标准名。
---

# Standard Translation

## 何时使用
1. 用户要求把已生成的 .md 草稿翻译成英文。
2. 已经存在 /workspace/output/drafts/<stem>.md。

## Instructions
1. 调用 write_file 写入 /workspace/output/drafts/<stem>.en.md。
2. 不得修改条款编号。
3. 不确定术语保留原文 + 译文括注。

## 失败处理
- 缺原文：先 task 委派 writer 生成中文草稿。

常见错误

  • frontmatter 缺 name 或 description：Deep Agents 无法发现该 skill。
  • description 过于笼统（如“帮助用户”）：主 Agent 难以判断何时加载；应具体描述触发条件。
  • 把大文件直接塞进 SKILL.md 正文：应放 references/，由 Agent 按需读取。

14. 如何自定义 Tool

本项目的工具都在 tools/__init__.py 集中注册。要新增一个工具，三步：

步骤 1：写工具文件

d:\deep-agents\src\standard_document_assistant\tools/translator.py
"""Translate a draft to English."""
from typing import Annotated, Any
from langchain.tools import ToolRuntime
from langchain_core.tools import InjectedToolArg, StructuredTool
from pydantic import BaseModel, Field

class TranslateDraftInput(BaseModel):
    file_path: str = Field(..., description="/workspace/output/drafts/*.md 虚拟路径")
    runtime: Annotated[ToolRuntime | None, InjectedToolArg] = None  # 不入 schema

def _translate_sync(file_path: str, *, runtime=None) -> dict[str, Any]:
    from standard_document_assistant.pathing import virtual_to_host_path
    src = virtual_to_host_path(file_path)
    target = src.with_suffix(".en.md")
    target.write_text(f"# English version of {src.stem}\n\n" + src.read_text(encoding="utf-8"),
                      encoding="utf-8")
    return {"virtual_path": f"/workspace/output/drafts/{target.name}", "host_path": str(target)}

translate_draft = StructuredTool.from_function(
    name="translate_draft",
    description="Translate a Markdown draft to English. Input: /workspace/output/drafts/*.md.",
    func=_translate_sync,
    args_schema=TranslateDraftInput,
)

步骤 2：在 tools/__init__.py 导出

from standard_document_assistant.tools.translator import translate_draft
# ...
STANDARD_DOCUMENT_TOOLS = [validate_output_schema, propose_memory_update, translate_draft]

步骤 3：把它挂到目标 SubAgent

# agent.py:build_subagents
{
    "name": "translator",
    "tools": [translate_draft, validate_output_schema],
    ...
}

常见错误

  • 把 runtime 写进 args_schema：会触发 PydanticInvalidForJsonSchema: core_schema.CallableSchema；必须用 InjectedToolArg 标记。
  • 未捕获异常：工具失败时模型只看到 ToolMessage 中的栈跟踪；本项目统一包成 {"status":"failed", "errors":[...]} 字典，行为等价于 handle_tool_errors=True。
  • 大结果直接返回：超过 ~16k token 会污染主 Agent 上下文；本项目落盘后只返回 virtual_path。

路径前缀readwrite设计意图
/workspace/.env*、含 secret 的文件 denydeny 硬性安全屏障：绝不让 Agent 触达凭据。
/workspace/input/**、/workspace/templates/** allowdeny 用户原始文件、模板：不可被 Agent 覆盖。
/workspace/output/**、/workspace/tmp/** allowallow 产物 / 草稿 / 临时文件可读写。
/skills/** allowdeny 技能只读；防止 Agent 篡改 skill 定义。
/memories/** allowdeny 长期记忆只读；更新走 propose_memory_update + HITL。

"""
ingest → retrieve_rules → judge_rules ─┐
                                       │ (Command)
                                  quality_gate ─┬─(widen)→ widen → reload → judge_rules (loop)
                                                 └─(ok)   → format_review → aggregate → write_outputs → write_manifest → END
"""
def quality_gate(state, runtime=None) -> Command[Literal["widen_review_scope", "format_review"]]:
    """Decide whether to widen the review scope and rerun the judge."""
    if (not state.get("format_only")
        and config.enable_llm_review
        and partial_mode != "full_document"
        and insufficient
        and round_idx < max_rounds):
        return Command(update=..., goto="widen_review_scope")
    return Command(update=..., goto="format_review")

19. Tracing / SSE

本项目通过 tracing.py 统一透传父 agent 的 callbacks / tags / metadata，子图以 standard_review / metadata_extraction 节点呈现在 LangSmith 树中。

  • 节点级 RetryPolicy：judge_rules / aggregate / run_langextract 对 transient 错误重试，避免对 LLM 解析错误无意义重试。
  • Tool 层流式事件：通过 ToolRuntime.stream_writer 推送 mineru.* / meta.* / review.* 命名空间事件，前端 stream_mode="custom" 消费。
  • 子图双通道 trace：state["trace_events"] 累加 + get_stream_writer 推送，二者通过同一节点函数发射。

20. FastAPI BFF（本地调试）

api/app.py 是面向 LangGraph Server 的本地 BFF：

  • 文件上传 → /workspace/input/uploads/{thread_id}/，返回 UploadedFileRecord。
  • 对话流式 → POST /threads/<id>/runs/stream，映射 LangGraph events → SSE 事件（sse_adapter.py）。
  • 中断恢复 → POST /threads/<id>/runs/<run_id>/resume 接收 Command(resume={"decisions": [...]})。
  • 产物下载 → /artifacts/<artifact_id>，artifacts.py 负责 host_path ↔ download_url 映射。
  • 直连审核 → POST /direct/review 绕过主 Agent，直接驱动 standard_review 子图（用于测试 / 异步审核）。

BFF 不修改业务逻辑，所有编排都来自 agent.py。

差异 / 注意点

  • 主 Agent 通过 task 工具把"读 PDF/Word/审核"委派给 SubAgent，避免上下文被大文本污染 —— 与官方 "Offload large results across long runs" 思路一致，但通过显式委派落地。
  • 技能 / 记忆权限统一收紧为 "read=allow / write=deny"，更新走工具 + HITL；这与官方 "Pause for human approval at critical decision points" 一致但更严格。
  • response_format 强制结构化输出，使 SSE / FastAPI BFF 渲染更稳定。
