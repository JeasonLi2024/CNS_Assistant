本文件结合 LangChain 官方 Deep Agents 文档（overview、customization、human-in-the-loop、context-engineering、managed-deep-agents） 对 d:\CNS_Assistant\src\standard_document_assistant 中 create_deep_agent 工厂的集成方式做系统性拆解，并辅以可视化图示。

1. 一句话概览

项目定位

标准文档助手（standard-document-assistant）是一个面向国标/行标 PDF/Word 的智能体， 覆盖 解析 元数据抽取 内容审核 格式审核 草稿生成 五大业务场景，基于 deepagents.create_deep_agent 构造。

Deep Agents 范式

官方把 Deep Agents 称为“agent harness”：以 LangGraph 的工具调用循环为底座， 内置 任务规划、文件系统、子代理、 长期记忆、HITL、上下文工程 6 大能力。 本项目正是把这套 harness 按 GB 标准审核业务“填实”。

2. Deep Agents 框架核心能力（官方视角）

① 任务规划 (Planning)

内置 todo 工具，复杂任务先拆解步骤再执行。

本项目由 MAIN_SYSTEM_PROMPT 强制要求主 Agent 在复杂任务前 调用 write_todos。

② 文件系统 (Filesystem)

Backend 抽象：State/Store/Filesystem/Composite。

本项目使用 CompositeBackend，将 /memories/、/skills/、 /workspace/ 三类虚拟路径分别路由到不同后端。

③ 上下文工程 (Context Engineering)

system prompt + memory + skills + middleware 多源组装。

本项目启动时加载 /memories/AGENTS.md、 /memories/preferences.md，技能目录按需注入。

④ 子代理 (Subagents)

隔离上下文、可并行、专业化分工。

本项目注册了 parser / extractor / reviewer / research / writer 5 个 SubAgent，分别负责不同业务链路。

⑤ HITL (Human-in-the-loop)

通过 interrupt_on 在敏感工具调用前暂停。

本项目对写文件、命令执行、记忆更新、MinerU 解析、审核等敏感工具配置了 ["approve", "edit"] 决策。

⑥ 长期记忆 (Memory)

StoreBackend + LangGraph Store，支持按用户/助手隔离 namespace。

本项目通过 seed_memory_store 启动注入 AGENTS.md / preferences.md / project-notes.md。

4. 主 Agent 编排（agent.py · build_standard_document_agent）

主 Agent 通过 create_deep_agent(**agent_kwargs) 一次性注入： 

# 来源：src/standard_document_assistant/agent.py
agent = create_deep_agent(
    name="standard-document-assistant",
    model=build_qwen_model(config.primary_model, strict=...),
    system_prompt=MAIN_SYSTEM_PROMPT,
    tools=STANDARD_DOCUMENT_TOOLS,                 # validate_output_schema, propose_memory_update
    memory=["/memories/AGENTS.md", "/memories/preferences.md"],
    skills=["/skills/"],
    subagents=build_subagents(...),
    backend=build_backend(...),
    permissions=build_permissions(),
    response_format=AgentResult,
    interrupt_on={                                # 主 Agent 直接调用层
        "write_file":           {"allowed_decisions": ["approve", "edit"]},
        "edit_file":            {"allowed_decisions": ["approve", "edit"]},
        "propose_memory_update":{"allowed_decisions": ["approve", "edit"]},
        "execute":              {"allowed_decisions": ["approve", "edit"]},
    },
    checkpointer=MemorySaver | None,               # langgraph_server=True 时由平台注入
    store=InMemoryStore | None,                      # 同上
)

关键设计点

  • 本地与部署双模式： langgraph_server 开关决定是否让平台托管 checkpointer / store。 本地默认 MemorySaver + InMemoryStore（注意：仅开发使用）。
  • 结构化输出： response_format=AgentResult 强制主 Agent 返回 summary / artifacts / quality_warnings / next_actions / download。
  • Skills 注入方式： skills=["/skills/"] 必须使用虚拟路径，Deep Agents 运行时会自动按需加载。
  • HITL 分层： 主 Agent 层只覆盖“主 Agent 自身直调”的工具；subagent 内的工具由 build_subagents 内部单独声明，避免重复。

6. Backend 路由（CompositeBackend）

官方建议在多源存储场景下用 CompositeBackend：把不同虚拟路径前缀路由到不同后端实现。 本项目 build_backend() 的路由表： 

CompositeBackend(
    default=StateBackend(),                                    # 短期 scratch / 大工具结果 offload
    routes={
        "/memories/":  StoreBackend(namespace=...),           # 长期记忆，按 assistant+user 隔离
        "/skills/":    FilesystemBackend(root=SKILLS_DIR, virtual_mode=True),
        "/workspace/": FilesystemBackend(root=WORKSPACE_ROOT, virtual_mode=True),  # 仅本机
    },
)

三条铁律（来自 agent.py 注释）

  • /memories/ 是虚拟路径；本地 memories/ 目录只用于 seed_memory_store 启动时初始化，运行时不会自动回写。
  • /workspace/ 仅在本地模式（langgraph_server=False）默认开启； 部署态默认关闭，必要时通过 STANDARD_DOC_ENABLE_WORKSPACE_BACKEND=1 打开。
  • StateBackend 作为默认后端，提供 thread-scoped 短期 VFS，避免污染长期记忆与真实磁盘。

7. 路径级权限矩阵（FilesystemPermission）

路径前缀readwrite设计意图
/workspace/.env*、含 secret 的文件 deny deny 硬性安全屏障：绝不让 Agent 触达凭据。
/workspace/input/**、/workspace/templates/** allow deny 用户原始文件、模板：不可被 Agent 覆盖。
/workspace/output/**、/workspace/tmp/** allow allow 产物 / 草稿 / 临时文件可读写。
/skills/** allow deny 技能只读；防止 Agent 篡改 skill 定义。
/memories/** allow deny 长期记忆只读；更新走 propose_memory_update + HITL。

8. HITL 审批策略（interrupt_on）

Deep Agents 官方 human-in-the-loop 通过 interrupt_on 在敏感工具调用前中断图执行，由人工 approve / edit 后再继续。

主 Agent 层（build_standard_document_agent）

{
  "write_file":            ["approve", "edit"],
  "edit_file":             ["approve", "edit"],
  "propose_memory_update": ["approve", "edit"],
  "execute":               ["approve", "edit"],
}

SubAgent 层（build_subagents）

# parser
{ "parse_file_with_mineru": ["approve", "edit"] }

# extractor
{ "extract_standard_metadata": ["approve", "edit"] }

# reviewer
{ "parse_file_with_mineru":        [...],
  "run_standard_review":         [...],
  "run_format_source_review":   [...],
  "build_review_index":          [...] }

开关与环境变量

  • hitl_enabled(langgraph_server=...) 同时支持 STANDARD_DOC_ENABLE_HITL=1 强制开启、 STANDARD_DOC_DISABLE_HITL=1 强制关闭。
  • langgraph dev 默认关闭 HITL，避免在 Studio 子图 resume 时触发 TypeError。
  • “reject” 显式留空：仅在严格禁用某能力的策略下才纳入白名单。

12. 与 Deep Agents 范式的对齐与差异

Deep Agents 官方能力本项目实现对标准文档助手的价值
任务规划 MAIN_SYSTEM_PROMPT 强制 write_todos，复杂任务先拆解 把“解析 → 抽取 → 审核 → 报告”拆成可追踪步骤，避免遗漏链路
Filesystem (Backend) CompositeBackend：/memories/ + /skills/ + /workspace/ + StateBackend 默认 把临时 scratch、长期记忆、领域知识、业务文件分到不同后端，治理清晰
Subagents 5 个专业 SubAgent，每个挂专属 Skill、工具与 HITL 配置 在隔离上下文里执行大文本解析/审核/草稿生成，主 Agent 只看摘要
Memory 启动注入 AGENTS.md / preferences.md；运行时走 StoreBackend；更新走 propose_memory_update 长期沉淀用户偏好与项目笔记，不可被 Agent 直写
Skills 4 个 skill 目录，virtual_mode=True 把 GB 标准审核知识与质量清单按需注入上下文，避免污染主 prompt
HITL interrupt_on 覆盖写文件、记忆更新、MinerU 解析、审核、构建索引 在覆写用户原文件、产物落盘、长期记忆更新前必须由人 approve/edit
Response Format response_format=AgentResult（pydantic） 统一摘要/产物/告警/下一步结构，下游 FastAPI BFF 容易消费
持久化 本地 MemorySaver + InMemoryStore；部署态由 LangGraph Server 注入 同一套工厂，langgraph_server 切换开发 / 生产
Managed Deep Agents（私有预览） 提供 managed_project_shape() 映射到项目结构 未来可零改造迁移到 LangSmith Managed Deep Agents

差异 / 注意点

  • 主 Agent 通过 task 工具把“读 PDF/Word/审核”委派给 SubAgent，避免上下文被大文本污染 —— 与官方 “Offload large results across long runs” 思路一致，但通过显式委派落地。
  • 技能 / 记忆权限统一收紧为 “read=allow / write=deny”，更新走工具 + HITL；这与官方 “Pause for human approval at critical decision points” 一致但更严格。
  • response_format 强制结构化输出，使 SSE / FastAPI BFF 渲染更稳定。
